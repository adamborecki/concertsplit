⏺ Issue #1 is a long product design doc written before any code was built. The
  core idea: a localhost Node.js webapp called "concertsplit" that lets
  classical music editors make editorial decisions in the browser — marking
  piece and movement boundaries, setting fades, handling applause — and then
  delegates all actual transcoding to FFmpeg.

  It argues the tool should be a decision-making surface, not a browser-based
  video editor or DAW. The recommended MVP was: local media preview, manual
  segment marking, piece/movement structure, metadata entry, fade settings, and
  JSON + FFmpeg command export.

  It also lists open questions like who the user is, whether sync is in scope
  for v1, and what the primary UX identity should be (cue sheet builder vs.
  timeline editor vs. deliverables manager).

  Most of this has since been built and shipped. Want to close it?

⏺ Good data. Here's my honest take on all three questions:

  ---
  Applause detection — do you need AI?

  Probably not a trained model. The current spectral flatness approach is the
  right idea, just needs tuning. With this labeled data you can run the detector
   against the KCA WAV, compare its output to your ground truth, and tune the
  threshold and duration parameters until it matches. That's faster and more
  reliable than training a model on 17 examples.

  Piece segmentation — do you need AI?

  This is harder. Piece boundaries are defined by musical content — silence,
  audience noise, tuning, applause patterns — and they vary a lot between
  concerts. A classical music model could learn this, but you'd need a lot of
  labeled concerts to train on. The more practical path near-term: use silence +
   applause detection together. Pieces almost always end with a sustained
  applause region preceded by a short silence. You can find those automatically
  without ML.

  If you do want AI eventually

  The right approach would be a fine-tuned audio classifier (e.g. on top of a
  pretrained model like CLAP or OpenAI Whisper's encoder) trained on labeled
  concert audio. But you'd need dozens of annotated concerts, not one.

  ---
  My recommendation:

  1. Now: use this labeled data to tune the existing detector — parse the txt,
  run FFmpeg spectral analysis on the KCA WAV, score the detector's output
  against ground truth, adjust thresholds
  2. Soon: add silence detection alongside applause to auto-suggest piece
  boundaries
  3. Later: revisit ML once you have 10+ labeled concerts
